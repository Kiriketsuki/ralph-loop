# Ralph Loop v2 — Parallel Code Review

> Generated by summon-reviewers — 5 specialist agents + 2 arbitration agents.
> Review scope: all changes in the v2 revamp (prompts/, stream/, agents.md, loop.sh, plan.sh, loop.ps1, plan.ps1, spec.md, CLAUDE.md).

---

## Critical / High Confidence

**[HIGH] plan-work completion detection always exits on iteration 1** (`loop.sh`, plan-work exit block)
The `grep -qE "\| *pending *\|"` check used to detect that the planning agent wrote new tasks will match *existing* pending rows already in `spec.md` from the initial plan. The loop will exit after iteration 1 every time, regardless of whether the plan-work agent actually wrote anything new. *(Raised independently by Agents 2, 4, 5)*

**[MEDIUM] envsubst substitutes `$WORK_SCOPE` inside markdown code blocks** (`loop.sh:132`, `prompts/plan-work.md`)
`envsubst` operates on raw text. `plan-work.md` contains the literal string `` `$WORK_SCOPE` `` in a backtick-quoted markdown span, which will be substituted and could be corrupted if `WORK_SCOPE` contains special characters or newlines. *(Raised by Agents 1, 4, 5)*

## Medium Confidence

**[HIGH] `mapfile` is bash 4.0+ only — breaks on macOS** (`stream/gutter.sh:41`)
`mapfile -t ENTRIES` is a bash 4.0 builtin. macOS ships with bash 3.2 (`/bin/bash`). On any macOS system where a user hasn't explicitly installed bash 4+, `gutter.sh` will fail silently with `mapfile: command not found`. Gutter detection is then skipped, and the loop will never detect stuck patterns. A version check or `while IFS= read -r` fallback is needed. *(Raised by Agents 4, 5)*

**[HIGH] Per-line `jq` process spawn for every Claude output line** (`stream/parser.sh:43-50`)
For `ENGINE=claude`, the parser spawns a `printf` subshell + a `jq` process for *every line* of stream-json output. A 64k-token Claude response can produce 15,000+ lines, resulting in 30,000+ child processes per iteration. At 2–5ms per invocation this adds 60–150 seconds of overhead per iteration. Fix: pipe the full stdin through a single `jq -R` streaming invocation rather than per-line. *(Raised by Agents 1, 3)*

**[HIGH] Gutter detection exits before git commit — leaves working tree dirty** (`loop.sh`, gutter block vs. git block ordering)
When gutter triggers (exit 4), the git add/commit/push block is never reached. The agent's file changes (updated `spec.md`, new `progress.md` entry) sit uncommitted in the working tree. A human resuming after exit 4 faces uncommitted state with no obvious recovery path. Recommend committing first, then running gutter detection, or documenting the manual recovery (`git add . && git stash`). *(Agent 5; both arbiters confirmed VALID)*

**[HIGH] PowerShell loop.ps1 lacks mode param, --dry-run, stream parser, gutter detection** (`loop.ps1`)
`loop.ps1` has no 5th `mode` argument, no `--dry-run` flag, pipes engine output directly to `Tee-Object` (bypassing `parser.sh` entirely — no token warnings, no rotation), and has no gutter detection call. Windows users get a functionally different (lesser) tool. The v2 plan explicitly scoped this as "minimal updates — paths only" so this is intentional, but it should be documented in CLAUDE.md as a known platform gap. *(Agent 5; both arbiters confirmed VALID)*

**[MEDIUM] Git commit message extraction has no fallback for uppercase type fields** (`loop.sh:250-252`)
The `sed` pattern `[a-z]*` requires a lowercase type from `progress.md`. If an agent writes `FEAT:` or omits the type, the extraction silently falls back to `chore(ralph): Iteration N automated progress sync` — a misleading commit message. The pattern could also fail if the progress entry contains embedded newlines (would corrupt the commit message). The existing fallback prevents breakage but masks agent errors. *(Raised by Agents 1, 2)*

**[MEDIUM] `specs/` empty directory is not tracked by git** (`specs/` directory)
Git does not track empty directories. When a user copies the ralph template to a new project, `specs/` will not exist. The first time an agent tries to create a topic spec, it will fail unless the directory already exists. A `.gitkeep` file is needed. *(Raised by Agents 4, 5)*

**[LOW] `LOG_FILE` parameter in `invoke_engine()` is redundant** (`loop.sh:169-171`)
`local LOG_FILE="$2"` is declared and immediately exported via `export LOG_FILE`, which exports the local value. Since the caller always passes the outer-scope `LOG_FILE` as the argument, the parameter is functionally redundant — removing it would give identical behaviour. The pattern creates false impression that the parameter affects something. *(Agent 4; both arbiters confirmed VALID)*

## Low Confidence (Arbitration Inconclusive)

**[MEDIUM] plan.sh plan-work mode lacks the branch guard that loop.sh has** (`plan.sh`, plan-work validation block)
`loop.sh` validates that plan-work mode requires a non-main/master branch. `plan.sh` has no equivalent. A user running `bash .ralph/plan.sh gemini "" plan-work` on main will silently proceed, potentially polluting the main spec with scoped tasks. *(Agent 5; arbiters split — both reasoning bodies confirm the code gap, verdicts differ)*

**[LOW] `parser.sh` comment says "Char count" but the env var is in tokens** (`stream/parser.sh:18`)
The comment on line 18 reads `RALPH_TOKEN_WARN - Char count for warning threshold` but the code multiplies the value by 4 and CLAUDE.md documents it as a token count. The comment is wrong, not the code. Low impact since the math is correct, but the contradictory comment will confuse anyone modifying thresholds. *(Agent 5; arbiters split on label, both reasoning bodies confirm the comment is wrong)*

---

## Summary

The v2 architecture is sound and the implementation is clean overall. Two issues need fixing before this is production-ready:

1. The plan-work completion detection will always false-positive exit on iteration 1.
2. `gutter.sh` will silently do nothing on macOS bash 3.2 (`mapfile` missing).

The per-line `jq` spawn is the highest-impact performance issue for long Claude sessions. The gutter-before-commit ordering is a UX rough edge worth resolving. Everything else is low-risk or intentionally scoped.
